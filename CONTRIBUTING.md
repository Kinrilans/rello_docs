# CONTRIBUTING — Rello Docs

Спасибо за вклад! Эти правила помогают держать документацию единообразной.

## 1) Как предлагать правки
- Делайте **ветку** от `main`: `docs/feature-<кратко>` или `docs/fix-<кратко>`.
- Вносите изменения в `.md` внутри соответствующей папки (`/product`, `/tech`, `/ops`, `/quality_launch`, `/bots`, `/legal`).
- Открывайте **Pull Request** в `main`. В описании укажите:
  - цель изменения,
  - связанные файлы,
  - затронутые версии (`_v2` → `_v2.1`, и т. п.).

## 2) Версионирование файлов
- Имя файла содержит версию, например: `api_contracts_v1.md`, `db_schema_v2.md`.
- **Минор** (редактура/уточнение): `_v2` → `_v2.1`.
- **Мажор** (существенные изменения структуры/объёма): `_v2` → `_v3` (старый файл не удаляйте, пометьте «архив» в заголовке).

## 3) Стиль и структура
- Язык: RU (можно EN для технических терминов).
- Заголовок `#` + короткое резюме в начале файла.
- Чёткие списки, таблицы, без длинных абзацев.
- Ссылки на смежные документы внизу файла («Связанные документы»).

## 4) Безопасность
- **Не коммить** секреты: токены, приватные ключи, `.env` (см. `.gitignore`). 
- Не публиковать персональные данные партнёров.
- Если правка связана с безопасностью, отметьте `security` в заголовке PR.

## 5) Чек-лист для PR
- [ ] Файл(ы) названы с корректной версией.
- [ ] Исправлены оглавления/ссылки, если затронуты.
- [ ] Нет секретов/учёток в диффе.
- [ ] Пройдена самопроверка орфографии/формата Markdown.

## 6) Роли
- Maintainer: владелец репозитория (слияние PR).
- Editor: оформление и вычитка.
- Author: добавляет/меняет содержимое.
